# Сценарий демонстрации проекта

## 1. Запуск системы

```bash
docker compose up -d --build
```

Проверить доступность:

- Frontend: http://localhost:3000
- Swagger: http://localhost:8000/docs
- Kafka UI: http://localhost:8080
- RedisInsight: http://localhost:5540

## 2. Проверка ролей

1. Войти под администратором:
   - `admin@example.com`
   - `admin12345`
2. Зарегистрировать обычного пользователя через `/api/v1/auth/register` или форму API.
3. Войти под пользователем в отдельной вкладке.
4. Убедиться, что пользователь видит только свои обращения и не видит журнал аудита.

## 3. Создание обращения

Пользователь создаёт обращение:

- тема: `Не отображается история обращений`;
- описание: `После входа в личный кабинет раздел «Мои обращения» пуст, хотя ранее были созданы обращения.`;
- приоритет: `MEDIUM`.

## 4. Обработка обращения администратором

1. Администратор видит обращение в общем списке.
2. Назначает исполнителя из списка активных администраторов.
3. Статус автоматически или вручную переводится в `IN_PROGRESS`.
4. Затем администратор переводит обращение в `RESOLVED`.
5. После подтверждения обращение переводится в `CLOSED`.

Недопустимый переход `OPEN → CLOSED` блокируется API.

## 5. Сообщения

1. Пользователь открывает карточку обращения и пишет сообщение.
2. Администратор отвечает.
3. В разделе «Сообщения» отображается диалог и счётчик непрочитанных сообщений.
4. При открытии диалога сообщения помечаются прочитанными.
5. После закрытия тикета новые сообщения не принимаются.

## 6. Kafka, уведомления и аудит

1. При создании, назначении, смене статуса и сообщениях backend публикует события в Kafka.
2. Notification Service создаёт персональные уведомления.
3. Audit Service сохраняет историю действий в `audit_logs`.
4. В Kafka UI можно посмотреть топики:
   - `ticket-events`;
   - `audit-events`.
5. В интерфейсе пользователь видит уведомления, а администратор — журнал аудита.

## 7. Redis

Статистика обращений кэшируется в Redis. Проверка через терминал:

```bash
docker compose exec redis redis-cli MONITOR
```

После обновления страницы будут видны команды `GET`, `SETEX`, `DEL`.

## Публичная demo-версия

Опубликованная версия доступна по адресам:

- Frontend: https://helpdesk-system-1.netlify.app
- Backend API: https://helpdesk-backend-vvws.onrender.com
- Swagger/OpenAPI: https://helpdesk-backend-vvws.onrender.com/docs

Для входа под супер-администратором:

- email: `admin@example.com`
- password: `Admin12345!`

Если пользователь ещё не зарегистрирован, нужно открыть форму регистрации в веб-интерфейсе и создать новую учётную запись. Такой пользователь автоматически получает роль `USER`, может создавать обращения и видеть только свои тикеты. Для проверки работы исполнителя супер-администратор может открыть управление пользователями и изменить роль нужной учётной записи на `AGENT`.
