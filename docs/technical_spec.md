# Техническое задание

## 1. Наименование проекта

«Система обработки обращений клиентов (HelpDesk System)».

## 2. Назначение системы

Система предназначена для регистрации, обработки и контроля обращений клиентов. Пользователь создаёт обращение, администратор назначает исполнителя, изменяет статус и отслеживает ход решения. Все ключевые действия журналируются, а события обработки обращений передаются в сервис уведомлений асинхронно через Apache Kafka.

## 3. Цели разработки

- автоматизировать приём и обработку клиентских обращений;
- обеспечить прозрачный жизненный цикл тикета;
- разграничить доступ пользователей по ролям;
- повысить производительность за счёт Redis-кэширования;
- разделить основную бизнес-логику, уведомления и аудит;
- подготовить контейнеризированную инфраструктуру для развёртывания.

## 4. Роли пользователей

| Роль | Возможности |
| --- | --- |
| USER | регистрация, вход, создание обращений, просмотр своих обращений, добавление сообщений |
| AGENT | просмотр назначенных обращений, изменение статусов, переписка с клиентом |
| SUPER_ADMIN | просмотр всех обращений, управление пользователями и ролями, назначение исполнителей, просмотр статистики и аудита |

## 5. Функциональные требования

1. Регистрация и аутентификация пользователей.
2. Выдача JWT-токена после успешного входа.
3. Управление пользователями администратором.
4. Создание обращения с темой, описанием и приоритетом.
5. Просмотр списка обращений с фильтрацией по статусу, исполнителю и текстовому запросу.
6. Просмотр карточки обращения.
7. Изменение статуса обращения.
8. Назначение исполнителя.
9. Добавление сообщений к обращению.
10. Формирование статистики по обращениям.
11. Публикация событий в Kafka при создании/изменении обращений и сообщений.
12. Обработка событий сервисом уведомлений с сохранением персональных уведомлений в БД.
13. Сохранение истории действий сервисом аудита и просмотр журнала администратором.
14. Кэширование статистики обращений в Redis.
15. Предоставление веб-интерфейса на React.

## 5.1. Жизненный цикл обращения

Система должна ограничивать переходы статусов:

| Текущий статус | Допустимый следующий статус |
| --- | --- |
| `OPEN` | `IN_PROGRESS` |
| `IN_PROGRESS` | `RESOLVED` |
| `RESOLVED` | `IN_PROGRESS`, `CLOSED` |
| `CLOSED` | нет переходов |

Обращение в статусе `CLOSED` не принимает новые сообщения и не может быть переназначено.
Исполнителем может быть только активный пользователь с ролью `AGENT`.
Суперадминистратор не участвует в клиентской переписке и получает только уведомления о новых обращениях, назначениях и изменениях статуса.

## 6. Нефункциональные требования

- REST API должен быть документирован через OpenAPI/Swagger.
- Сервисы должны запускаться через Docker Compose.
- Данные должны храниться в PostgreSQL.
- Система должна поддерживать горизонтальное развитие за счёт выделения сервисов уведомлений и аудита.
- Доступ к данным должен проверяться на уровне API.
- Код должен быть покрыт базовыми модульными и интеграционными тестами.

## 7. Ограничения проекта

В учебной реализации уведомления сохраняются в PostgreSQL и отображаются в веб-интерфейсе. Дополнительно поддержаны внешние каналы: Mailtrap Sandbox для безопасной демонстрации email без собственного домена и Telegram Bot API для отправки сообщений пользователям, указавшим `chat_id`.
